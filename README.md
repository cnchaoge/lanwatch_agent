# lanwatch_agent - 企业网络值守服务

> 让每一家中小企业的网络都有"人"值守

[![版本](https://img.shields.io/badge/version-v0.5.0-blue.svg)](https://github.com/cnchaoge/lanwatch_agent/releases)
[![平台](https://img.shields.io/badge/platform-Windows%20%7C%20Linux%20%7C%20macOS-green.svg)]()
[![协议](https://img.shields.io/badge/license-MIT-orange.svg)]()

**lanwatch_agent** 是一套面向中小企业网络监控的 SaaS 解决方案，包含运行在客户内网的监控客户端（Agent）和部署在云端的服务端。客户无需专业人员，下载运行即可完成部署，管理员和企业主通过 Web 后台或手机随时掌握网络健康状态。

---

## 🎯 解决什么问题

三四线城市的中小企业普遍面临网络故障发现滞后的问题——往往是业务受损后才报修，运维响应被动。**lanwatch_agent** 将监控能力打包成"网络值守服务"：企业安装一个 exe，服务端持续采集数据，离线即报警，手机随时查看。

---

## ✨ 核心功能

| 功能 | 说明 |
|------|------|
| **多目标监控** | 可同时监控网关、DNS、ERP 等多个目标，支持自定义 IP 或域名 |
| **内网设备发现** | 每 5 分钟自动扫描局域网，发现所有在线设备（IP / MAC / 厂商 / 设备类型） |
| **系统托盘运行** | 最小化到托盘，不占任务栏，后台静默运行 |
| **开机自启** | 支持注册表自启，客户开机即监控，无需人工干预 |
| **微信报警** | 网络离线超过 3 分钟自动通过 Server酱 推送微信通知 |
| **实时状态指示** | 托盘图标颜色直观显示网络健康状态（绿=正常，红=故障） |
| **手机端查看** | 适配手机浏览器，无需安装 App，扫码即看 |
| **多租户隔离** | 每个企业主只能看到自己公司的数据，完全隔离 |
| **SNMP 监控** | 支持企业级路由器/交换机的 SNMP 协议接入（V2c） |
| **SNMP Trap 接收** | 被动接收设备主动上报告警，无需轮询 |

```
│  └─────────────────────────┘  └────────────────────┘          │

```

**数据流说明：**

1. Agent 启动 → 自注册获取 `agent_id` → 每 60 秒上报探测数据
2. 服务端写入 SQLite，并通过 SSE 实时推送到所有在线浏览器
3. 离线超过 3 分钟 → Server酱 触发微信报警
4. 每 5 分钟 Agent 在后台线程扫描局域网拓扑上报

## ⚡ 技术亮点：为什么用 ICMP Ping 而不是 TCP 端口探测

> 这是 v0.5 与旧版最重要的技术差异

### 传统方案的问题：TCP 端口探测（v0.4 及之前版本）

旧版客户端通过 **TCP 连接 80/443 端口** 判断设备是否在线：

```python
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect_ex((host, 80))   # 只有开了 HTTP 服务才响应
```

**这会带来严重的漏报问题：**

| 设备类型 | 典型情况 | 结果 |
|----------|----------|------|
| 打印机（网络打印共享） | 通常端口 9100（JetDirect） | ❌ 误报离线 |
| IP 摄像头 | 自有协议，不开 Web 服务 | ❌ 误报离线 |
| IoT 设备（摄像头、门禁） | 自定义协议，无 HTTP | ❌ 误报离线 |
| 工业设备、PLC | 仅开放专有端口 | ❌ 误报离线 |
| 部分路由器管理界面 | 可能关闭了 HTTP | ❌ 误报离线 |

本质上，**"端口可达"≠"设备在线"**。TCP 端口探测只适合探测开了对应服务的服务器。

### lanwatch_agent v0.5 的方案：ICMP Ping

ICMP ping 是网络层的探测协议，任何设备只要 IP 活着、网卡在运作，就会响应 ICMP（除非被防火墙主动屏蔽），与它运行了什么服务无关：

```python
# Windows
subprocess.run('ping -n 1 -w 300 192.168.1.1', ...)

# Linux
subprocess.run('ping -c 1 -W 1 192.168.1.1', ...)
```

**对比效果：**

| 对比维度 | TCP 80 端口探测 | ICMP Ping |
|----------|-----------------|-----------|
| 探测覆盖率 | ~30%（仅开 Web 服务的设备） | ~95%（几乎所有 IP 设备） |
| 对网关的探测 | 依赖网关开了 Web 管理界面 | ✅ 任何网关都响应 ICMP |
| 对打印机的探测 | ❌ 端口 9100 不通 | ✅ 响应 ICMP |
| 对摄像头的探测 | ❌ 通常无 HTTP | ✅ 响应 ICMP |
| 对工业设备的探测 | ❌ 协议未知 | ✅ 响应 ICMP |
| 网络层语义 | 应用层（可被防火墙拦） | 网络层（标准 ICMP 协议） |

### MAC 地址获取的正确姿势

TCP 探测方案还有一个致命缺陷：即使端口通了，也拿不到设备的 MAC 地址（TCP 不暴露链路层信息）。

`arp_lookup` 的正确流程是：

```
1. ping -n 1 <目标 IP>        ← 先发 ICMP，触发对方 ARP 缓存记录
2. arp -a <目标 IP>            ← 再查 ARP 表，此时表里有对应条目
```

旧版直接发 UDP 空包到 80 端口，ARP 表里根本不会有记录。

### 拓扑扫描性能：串行 vs 多线程

| 方案 | 254 IP 扫描耗时 | CPU 利用率 |
|------|----------------|-----------|
| 串行 TCP 探测（每次 0.5s 超时） | ~127 秒 | 低（大量时间在等网络） |
| 串行 ICMP Ping | ~85 秒 | 低 |
| **多线程 ICMP Ping（30 workers）** | **~5 秒** | 高效 |

v0.5 采用 `ThreadPoolExecutor(max_workers=30)` 并发探测，充分利用网络等待间隙，254 个 IP 从 2 分钟缩短到 5 秒以内。

## 📦 安装部署

### 企业主（客户端）

**Windows 版本（推荐）：**

1. 下载 `lanwatch_agent.exe`：https://github.com/cnchaoge/lanwatch_agent/releases/latest
2. 双击运行，填写企业名称
3. 首次运行后可勾选「开机自动启动」
4. 最小化到托盘，开始监控

**服务端地址：** `http://82.156.229.67:8000`

**手机查看：** http://82.156.229.67:8000/mobile

---

### 开发者（从源码构建）

**依赖：** Python 3.8+

```bash
# 克隆仓库
git clone https://github.com/cnchaoge/lanwatch_agent.git
cd lanwatch_agent/agent

# 安装依赖
pip install pystray pillow pyinstaller

# 打包 Windows exe（需要 Windows 系统）
build.bat

# 生成的 exe 在 dist/lanwatch_agent.exe
```

---

### 私有部署（服务端）

服务端依赖 Python 3.8+ 和 FastAPI：

```bash
cd server
pip install -r requirements.txt
python main.py
# 服务监听 0.0.0.0:8000
```

**服务端 requirements.txt：**
```
fastapi uvicorn python-multipart jinja2 aiofiles
pysnmp  # 可选，SNMP 监控需要
qrcode   # 二维码生成
```
